Executive Summary
This document presents deep technical research for building a custom DFIR (Digital Forensics and Incident Response) Agent Builder tool -- a purpose-built alternative to Velociraptor's offline collector that packages all of its power into a clean, modern UI specifically designed for mass endpoint deployment via GPO. The tool enables IR engineers to visually select forensic artifacts, configure AWS S3 upload targets, and compile a single hardened Windows executable -- ready to push to thousands of endpoints.

The research covers three major pillars: (1) the internal mechanics of Velociraptor's collector binary, including its config-repack architecture, artifact selection engine, encryption stack, and VQL execution model; (2) the complete catalog of forensic artifacts available for collection mapped to their IR value; and (3) AWS security architecture for write-only, tamper-evident forensic evidence collection including IAM policies, S3 Object Lock, KMS encryption, and credential embedding strategies.

A fourth pillar -- the complete technical blueprint for the Agent Builder tool itself -- is covered in Part V and VI, including the full UI flow, backend build engine, GPO deployment strategy, and implementation roadmap. This document is intended to serve as the primary planning reference for the project.

Part I -- How Velociraptor's Collector Builder Works Internally
1.1 The Core Architecture: Config-Repack Model
Velociraptor's offline collector is not a separately compiled binary -- it is the standard Velociraptor binary with a compressed, embedded YAML configuration injected into a reserved section of the PE (Portable Executable). This is the config-repack architecture and it is the single most important design insight for your custom tool.

How it works step by step:
• The base Velociraptor binary (~20MB) contains a fixed reserved embedding section (~80KB compressed).
• When you run config repack, the tool deflate-compresses a YAML spec into that section, overwriting placeholder bytes.
• On execution without arguments, the binary self-reads its embedding section, decompresses the config, and enters autoexec mode.
• The autoexec config contains: artifact names, artifact VQL definitions (custom ones only -- built-ins are compiled in), tool binaries (e.g. winpmem for memory), and output/upload settings.
• Digital signature is intentionally invalidated by the repack -- Velociraptor documents this and provides the Generic Collector format as an alternative for macOS/strict environments.

# Extract config from existing collector:
Collector.exe config show > config.yaml

# Repack modified config back into binary:
velociraptor.exe config repack config.yaml new_collector.exe

# CLI-only build (no GUI required, v0.75.5+):
velociraptor.exe collector --config server.config.yaml --spec collector_spec.yaml output_collector.exe

1.3 VQL Execution Model
Understanding VQL (Velociraptor Query Language) is essential for knowing why the collector is so powerful and flexible:
• VQL is a SQL-inspired query language where each query produces a stream of rows (JSON objects).
• Artifacts are named, parameterized VQL programs -- they accept inputs (parameters) and produce output rows.
• The collector runs each selected artifact's VQL in sequence, serializing output to JSONL or CSV format within the evidence zip.
• Built-in artifacts (prefetch, MFT, etc.) are compiled into the Go binary -- they do NOT add to the config embed size.
• Custom/exchange artifacts are embedded as VQL text in the config -- these DO count against the ~80KB limit.
• Each artifact can spawn sub-queries using foreach(), file(), glob(), and plugin calls to external binaries.
• The Windows.KapeFiles.Targets artifact is a special VQL wrapper that uses KAPE-style target definitions (YAML) to identify and copy raw files -- it can collect hundreds of file categories through a single artifact call.

VQL Plugin Architecture (key plugins used in IR artifacts):
• glob() -- file path pattern matching with VSS snapshot support
• stat() -- file metadata (timestamps, size, attributes)
• read_file() -- raw file reading with offset/length support
• parse_mft() -- direct NTFS MFT parsing without filesystem abstraction
• parse_usn() -- USN journal parsing
• parse_prefetch() -- Windows Prefetch binary parsing
• parse_evtx() -- Windows Event Log EVTX parsing
• reg_key() / reg_values() -- Windows Registry reading
• upload_s3() -- direct S3 PutObject calls
• upload_sftp() -- SFTP upload with embedded key

1.4 Artifact Embedding and the ~80KB Limit
This limit is critical for your tool's design. The ~80KB compressed space inside the PE binary constrains how many artifact definitions can be embedded. Key insights:
• Built-in artifacts (compiled into the binary) do NOT count against this limit -- only custom or 3rd-party artifact definitions.
• Compression ratio is very high for YAML (~10:1), so the uncompressed equivalent is ~800KB of YAML definitions.
• Large artifact bundles (KapeFiles, Memory acquisition with winpmem) can breach this limit when combined.
• Solution: Use the Generic Collector format -- a separate .collector file alongside the binary, with no size limit.
• Your builder UI should warn when estimated config size approaches the limit and auto-suggest Generic Collector mode.

1.5 Tool Embedding (3rd Party Binaries)
Velociraptor can bundle external executables (e.g., winpmem for memory, autoruns, sigcheck) directly into the collector binary. These are stored in an inventory manifest and extracted at runtime. For your builder:
• Each tool adds to file size -- winpmem alone is ~1.5MB for 64-bit.
• Tools are verified via SHA256 hash at extraction time (chain of custody).
• Your UI should show tool size impact as users add memory/external tool artifacts.
• Tools are defined in Server.Internal.ToolDependencies -- your builder should reference this manifest.
• winpmem v4+ supports modern Windows 10/11 and Server 2019/2022 -- always use the latest stable release.
• Autoruns from Sysinternals can be embedded and run at collection time to enumerate all persistence mechanisms.

3.3 Complete AWS Infrastructure Setup
The following outlines every AWS resource that must be created for a production-grade forensic evidence collection infrastructure:

STEP 1: AWS Account StructureCreate a dedicated Forensics AWS Account via AWS Organizations (separate from production).Enable AWS CloudTrail with multi-region trail in this account -- log all API calls.Enable AWS Config for compliance monitoring of S3 bucket settings.Set up AWS GuardDuty to detect anomalous API calls on the evidence bucket (e.g., GetObject from unknown IP).Enable S3 Block Public Access at the account level -- forensic buckets must never be public.
STEP 2: S3 Evidence Bucket ConfigurationBucket name: ir-evidence-[orgname]-[year] (e.g., ir-evidence-acmecorp-2026)Region: ap-south-1 (Mumbai) for India-based IR teams, or match where endpoints are deployedVersioning: ENABLED -- protects against accidental overwrites, required for Object LockObject Lock: ENABLED, Compliance mode, default retention 365 days (cannot be overridden even by root)Server-side encryption: SSE-KMS with a Customer Managed Key (CMK) -- NOT SSE-S3Bucket policy: Deny any non-HTTPS access (aws:SecureTransport condition)Bucket policy: Deny s3:DeleteObject from ALL principals including rootLifecycle policy: Transition to S3 Glacier after 90 days (cost optimization)Notification: S3 Event Notification -> SNS -> IR team email on any new PutObject (real-time alert)
STEP 3: KMS Customer Managed Key SetupCreate a CMK in ap-south-1 with key material type: KMS (AWS managed) or EXTERNAL (bring your own)Key policy: Key administrators = IR team IAM roles (can manage but not use for decryption)Key policy: Key users = Collector IAM user (kms:GenerateDataKey only) + Analyst IAM role (kms:Decrypt)Enable automatic key rotation (annually) -- protects against long-term key exposureEnable CloudTrail logging for all KMS API calls (kms:Decrypt calls = who accessed evidence)Multi-Region Key (MRK): Required if collecting from endpoints in multiple AWS regions
STEP 4: IAM User for Collector (Embedded in EXE)Create IAM User: velociraptor-collector-[buildid] (unique per collector build)Attach write-only policy (see below) -- literally the only permission this user hasCreate Access Key for this user -- this is what gets embedded in the collector binarySet password policy: No console access (programmatic-only user)Set access key expiry: 90 days maximum -- rebuild collectors on rotation scheduleTag the user with: build-id, build-date, artifact-set, created-by for audit tracking
STEP 5: IAM Role for IR Analysts (Never Embedded)Create IAM Role: ir-analyst-evidence-reader (assumed by analysts, never in any binary)Attach read-only policy: s3:GetObject, s3:ListBucket, s3:GetObjectVersionAttach KMS decrypt policy: kms:Decrypt, kms:DescribeKey for the evidence CMK onlyRequire MFA for role assumption (MFA condition in trust policy)Session duration: 4 hours maximum (collectors run < 1 hour, analysts need longer)Require source IP condition: restrict assumption to corporate VPN IPs or AWS SSO

3.5 AWS SFTP via Transfer Family (Maximum Security Option)
For environments requiring the highest security, AWS Transfer Family SFTP backed by S3 provides write-only access where the embedded credential is an SSH private key -- not AWS credentials. Even full binary disassembly reveals no AWS credentials.
• Write-only by policy: The SFTP user has a scope-down IAM policy that allows only s3:PutObject.
• The collector binary embeds only an SSH private key -- not AWS access keys.
• Even if the binary is fully reverse-engineered, the attacker can only upload new files.
• Different SFTP users can be created for investigators (GetObject) vs. collectors (PutObject only).
• Transfer Family logs all SFTP sessions to CloudWatch -- full audit trail of every upload.

# SFTP Target Configuration:
target: SFTP
target_args:
  upload_file: /evidence/Collection-%FQDN%-%TIMESTAMP%.zip
  sftp_server: s-xxxx.server.transfer.ap-south-1.amazonaws.com
  sftp_user: velociraptor-upload-user
  sftp_private_key: |  # Ed25519 private key embedded here
    -----BEGIN OPENSSH PRIVATE KEY-----
    [key data]
    -----END OPENSSH PRIVATE KEY-----

3.6 S3 Object Naming Strategy (Preventing Collisions)
The S3 object key (path) structure must be designed to prevent collisions when thousands of endpoints upload simultaneously:

# Recommended S3 Object Key Template:
{IAM_USERNAME}/{SITE_CODE}/{HOSTNAME}/{TIMESTAMP_ISO8601}_{UUID4}_collection.zip

# Example:
velociraptor-collector-build007/APAC-HYD/LAPTOP-ABC123/2026-05-06T14-30-00Z_a3f9b221_collection.zip

• IAM_USERNAME prefix: enforced by IAM policy -- collector can only write to its own prefix.
• SITE_CODE: logical grouping for bulk analysis (APAC-HYD, EU-LON, US-NYC etc.).
• HOSTNAME: %FQDN% Velociraptor variable -- fully qualified domain name of endpoint.
• TIMESTAMP_ISO8601: %TIMESTAMP% formatted as sortable ISO 8601 string.
• UUID4 suffix: prevents clock-skew collisions on same host within same second.

Part IV -- Encryption Architecture
4.1 Velociraptor Encryption Options

Encryption Scheme ComparisonPASSWORD: Simple symmetric AES password. Weakest -- password must be embedded in binary or provided at runtime.X509 (RECOMMENDED): Asymmetric. RSA/ECC public key embedded in collector -- only the private key can decrypt. Even full binary disassembly reveals no decryption capability.PGP: Asymmetric like X509. Random session key encrypted with PGP public key. Equivalent security to X509 but does not support auto-import to Velociraptor server.NONE: No container encryption. Still protected by TLS in transit and SSE-KMS at rest -- acceptable if S3 controls are strong.
4.2 X509 Encryption Deep Dive
X509 is the recommended and most practical scheme for enterprise IR:
• Collector generates a cryptographically random AES-256 session key at runtime (different for every collection).
• Session key is encrypted with the RSA/ECC public key and stored in the zip container header.
• The zip file itself is AES-256 encrypted with the session key.
• Only the private key holder (the IR analyst's system or Velociraptor server) can decrypt.
• Compromising the binary only reveals the public key -- attacker can upload but never read evidence.
• Auto-import: When the server's own certificate is used, Velociraptor automatically decrypts on import.

Key generation for your custom tool:
# Generate RSA-4096 key pair (Python cryptography library):
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import serialization

private_key = rsa.generate_private_key(public_exponent=65537, key_size=4096)
public_key = private_key.public_key()

# Export public key as PEM (embed in collector):
pub_pem = public_key.public_bytes(
    serialization.Encoding.PEM,
    serialization.PublicFormat.SubjectPublicKeyInfo
)

# Store private key in AWS Secrets Manager:
# aws secretsmanager create-secret --name dfir/build-007/private-key \
#   --secret-string file://private_key.pem

4.3 Transport Encryption (In-Transit)
• All S3 uploads enforce HTTPS (TLS 1.2 minimum via AWS policy) -- use the condition aws:SecureTransport: true in IAM policies.
• SFTP is inherently encrypted via SSH transport layer.
• The container is also encrypted before upload -- so evidence is encrypted at rest AND in transit (double encryption for the paranoid).
• AWS S3 provides SSE-KMS server-side encryption as a third layer when configured.
• Three layers in total: X509 container encryption + TLS in-transit + SSE-KMS at-rest.

5.3 Backend Build Pipeline
The sequence of operations the backend performs when the user clicks BUILD:

# Backend Build Pipeline (Python pseudocode)
def build_collector(spec: CollectorSpec) -> BuildResult:
    build_id = generate_uuid()

    # Step 1: Generate encryption keys
    if spec.encryption == 'X509':
        private_key, public_cert = generate_x509_keypair(key_size=4096)
        vault_store(f'dfir/{build_id}/privkey', private_key)
        spec.encryption_args.public_key = public_cert

    # Step 2: Generate collector_spec.yaml
    spec_yaml = render_spec_yaml(spec, build_id)
    write_temp(f'/tmp/{build_id}/collector_spec.yaml', spec_yaml)

    # Step 3: Estimate config size and choose embed mode
    compressed_size = estimate_compressed_size(spec_yaml)
    use_generic = compressed_size > 60 * 1024  # 60KB threshold

    # Step 4: Invoke Velociraptor CLI
    result = subprocess.run([
        'velociraptor.exe', 'collector',
        '--config', 'server.config.yaml',
        '--spec', f'/tmp/{build_id}/collector_spec.yaml',
        f'/tmp/{build_id}/Collector_{build_id}.exe'
    ], capture_output=True, timeout=300)

    # Step 5: Optional code signing
    if config.SIGNING_CERT_PATH:
        signtool_sign(f'/tmp/{build_id}/Collector_{build_id}.exe')

    # Step 6: Write audit log
    audit_log.write(build_id, spec, public_cert_fingerprint)

    return BuildResult(exe_path=..., build_id=build_id, metadata=...)

5.4 Collector Binary Security Hardening
• Embed only the write-only IAM access key -- use AWS STS to generate scoped-down temporary credentials if possible.
• Use a dedicated IAM user per build (unique access key per collector version) -- makes it easy to revoke a specific collector's upload capability.
• Set IAM access key expiry (90 days max) -- rotate embedded keys by rebuilding collectors regularly.
• The opt_delete_after: Y setting removes the local zip after successful upload -- reduces on-disk exposure.
• The opt_require_admin: Y setting ensures the collector fails gracefully if run without elevation.
• Consider code signing the output exe with a corporate code signing certificate to prevent EDR blocks.
• The X509 encrypted container means even the evidence-handling team needs the private key -- build an access control for it.
• Consider process protection: rename collector to something innocuous for stealth deployment if needed.

5.5 GPO Deployment Strategy
• Deploy the collector EXE via GPO Startup Script (runs as SYSTEM -- satisfies admin requirement).
• OR deploy via GPO Software Installation as a dummy MSI wrapping the EXE.
• OR use SCCM/Intune for modern device management -- push as a Win32 App.
• Use the %FQDN%-%TIMESTAMP% naming template so each endpoint creates a uniquely named S3 object.
• Use S3 prefix per site: s3://bucket/{SITE}/{HOSTNAME}/{TIMESTAMP}_{UUID}.zip
• The opt_prompt: N setting ensures the collector runs silently -- no user interaction required.
• The collector creates its own temp directory, runs, uploads, optionally deletes local evidence, and exits.
• Total runtime for QuickTriage: 5-15 minutes. For SANSTriage: 30-60 minutes. Design GPO timeout accordingly.
• For SCCM: Deploy as 'Required' with detection rule: registry key written by collector on completion.
• For Intune: Use PowerShell wrapper script as Win32 app, use file existence as detection rule.
